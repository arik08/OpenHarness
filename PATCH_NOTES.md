## 2026-04-25 (Myeongcheol)

이 날의 가장 큰 변화는 OpenHarness가 처음으로 웹 앱의 형태를 갖추기 시작했다는 점입니다. 기존에는 명령줄과 터미널 UI에서 쓰는 도구에 가까웠기 때문에, 대화 기록을 다시 찾거나 생성된 파일을 확인하거나 설정을 바꾸는 일이 화면 안에서 매끄럽게 이어지지는 않았습니다. 그래서 브라우저에서 OpenHarness를 사용할 수 있는 첫 웹 UI를 만들었습니다.

- 웹 화면에서 메시지를 입력하면 실제 OpenHarness 작업으로 이어지고, 응답이 다시 브라우저에 표시되도록 연결했습니다. 단순한 화면 시안이 아니라, 실제 에이전트 실행과 이어지는 첫 구조였습니다.
- 처음 웹 UI를 만들 때부터 단순히 기능을 붙이는 데서 그치지 않고, CLI 도구를 웹 앱으로 옮겼을 때 어색하지 않도록 레이아웃과 시각적 밀도를 함께 잡기 시작했습니다. 대화 영역, 사이드바, 입력창, 보조 패널이 한 화면 안에서 서로 밀어내지 않고 공존하는 것이 중요한 기준이었습니다.
- 대화 기록을 웹에서 보고 다시 선택할 수 있는 기반을 넣었습니다. 터미널 중심 사용에서는 지난 작업을 되찾는 일이 불편했기 때문에, 웹 UI에서는 히스토리를 중요한 첫 기능으로 잡았습니다.
- 에이전트가 만든 파일이나 이미지, 문서 결과물을 따로 확인할 수 있도록 아티팩트 영역의 기초를 만들었습니다. 이후 결과물을 브라우저에서 바로 열어보는 기능들이 여기에 이어졌습니다.
- 프로젝트 안의 파일을 경로로 직접 입력하는 대신, 웹 화면에서 고를 수 있는 흐름을 추가했습니다. 웹 UI를 만든 목적이 “덜 외우고, 더 보고 고르는 방식”이었기 때문에 필요한 작업이었습니다.
- Windows에서 웹 UI를 쉽게 켤 수 있도록 실행 파일을 추가했습니다. 개발자용 명령을 매번 조합하지 않고도 로컬 웹 화면을 열 수 있게 하려는 목적이었습니다.
- 웹 세션을 오래 켜 두고 쓰는 상황을 고려해, 세션이 끝날 때 남는 연결과 상태를 정리하는 부분도 함께 보강했습니다.
- 모델을 고를 때 최신 OpenAI 계열 기본값과 추론 강도 옵션을 쓸 수 있도록 정리했습니다. 웹 UI에서 설정을 다루기 시작했기 때문에, 모델 선택 경험도 같이 맞춰야 했습니다.
- 스킬 사용 이력이 사이드바와 채팅에서 더 자연스럽게 보이도록 고쳤습니다. 사용자가 “지금 어떤 보조 기능이 쓰였는지”를 화면에서 이해할 수 있게 하려는 변경입니다.
- 프로젝트 삭제는 실수하면 피해가 크기 때문에, 바로 삭제하지 않고 한 번 더 확인하는 흐름으로 바꿨습니다.
- 브라우저 점검 중 생기는 임시 산출물이 Git 변경 목록에 섞이지 않도록 Playwright MCP 관련 ignore 규칙도 정리했습니다.

## 2026-04-26 (Myeongcheol)

전날 만든 웹 UI를 “켜지는 화면”에서 “실제로 오래 작업할 수 있는 화면”으로 끌어올리는 데 집중한 날입니다. 대화, 히스토리, 아티팩트, 설정, 세션 관리가 한 화면 안에서 계속 이어져야 했기 때문에 사용 흐름과 안정성을 같이 손봤습니다.

- 도움말과 Plan Mode 표시를 정리했습니다. 사용자가 지금 명령을 실행하는 중인지, 먼저 계획을 세우는 중인지 헷갈리지 않게 하고, 화면 안에서 할 수 있는 일을 더 쉽게 파악하게 하려는 목적이었습니다.
- 화면 밀도와 테마를 다듬었습니다. 기능이 빠르게 늘어나면서 화면이 답답해질 수 있었기 때문에, 버튼과 패널 간격, 선택지 모양, 텍스트 크기를 더 오래 보기 편한 쪽으로 조정했습니다.
- 디자인적으로는 “개발 도구지만 너무 터미널스럽지 않고, 그렇다고 과한 대시보드처럼 보이지 않는” 균형을 잡는 데 시간을 썼습니다. 패널을 큼직한 카드처럼 띄우기보다 정보가 조밀하게 정리되도록 하고, 버튼과 라벨은 작지만 눌러야 할 위치가 분명하게 보이도록 맞췄습니다.
- 사이드바, 대화 영역, 보조 패널이 동시에 보이는 상황에서도 시선이 흩어지지 않게 레이아웃을 조정했습니다. 특히 히스토리와 아티팩트처럼 자주 열고 닫는 영역은 화면을 크게 흔들지 않도록 다듬었습니다.
- 여러 세션을 한꺼번에 열어도 화면과 백엔드가 감당할 수 있도록 동시 세션 수를 제한했습니다. 무작정 많이 열리는 것보다, 사용자가 상태를 이해할 수 있는 범위 안에서 관리되게 한 것입니다.
- 위험한 명령에 대한 안내를 더 분명하게 바꿨습니다. 웹에서는 클릭 한 번으로 작업이 이어질 수 있으므로, 삭제나 덮어쓰기 같은 작업은 사용자가 위험을 인지한 뒤 진행하도록 했습니다.
- 여러 대화 흐름을 나눠 볼 수 있는 세 방향 채팅 슬롯을 추가했습니다. 하나의 대화에 모든 실험과 참고 내용을 섞지 않고, 작업 흐름을 분리해 볼 수 있게 하려는 기능입니다.
- 파일 변경이 동시에 겹칠 때 생길 수 있는 충돌을 줄이기 위해, 변경 작업이 진행 중이면 다른 변경이 기다리도록 하는 안전장치를 보강했습니다.
- 히스토리 패널을 더 자연스럽게 다듬었습니다. 과거 대화를 불러오고, 선택하고, 삭제하는 동작이 현재 작업 중인 채팅을 불필요하게 흔들지 않도록 고쳤습니다.
- 아티팩트 패널을 본격적으로 사용 가능한 형태로 정리했습니다. 생성 결과물을 열어보는 것뿐 아니라 원본 내용까지 확인할 수 있게 해서, 결과물이 어떻게 만들어졌는지 추적하기 쉬워졌습니다.
- 답변 아래의 액션 버튼과 스트리밍 상태 표시를 정리했습니다. 답변이 생성 중인지 완료됐는지, 다음에 무엇을 할 수 있는지 더 분명히 보이게 하려는 작업이었습니다.
- 웹 자산 패키징을 정리했습니다. 초기 작업 중 설치 산출물이 과하게 섞였기 때문에, 실제 실행과 배포에 필요한 파일 중심으로 남기고 불필요한 파일은 걷어냈습니다.
- Windows 환경에서 웹 UI를 쓰는 경우를 중요하게 보고, 경로 처리, 실행 파일, 콘솔 동작, 테스트를 보강했습니다.
- 프로젝트마다 다른 P-GPT 설정을 둘 수 있게 했습니다. 회사 내부용, 개인용, 테스트용 연결을 같은 전역 설정에 억지로 섞지 않기 위한 변경입니다.
- JSON 기반 로그인과 자격 증명 예시를 추가했습니다. 설정값을 매번 손으로 입력하는 대신, 정리된 파일을 보고 맞춰 넣을 수 있게 하려는 목적이었습니다.
- 웹 설정에서 고른 값이 실제 작업 세션까지 전달되도록 보강했습니다. 화면에서 설정한 내용과 에이전트가 실제로 쓰는 환경이 어긋나지 않게 하는 데 초점을 맞췄습니다.
- 채팅 히스토리의 이름과 오류 표시를 다듬었습니다. 대화가 많아질수록 “어떤 대화였는지”와 “무엇이 실패했는지”를 바로 이해하는 것이 중요해졌기 때문입니다.
- workspace 기준으로 살아 있는 세션을 구분할 수 있게 했습니다. 프로젝트별로 작업 맥락을 나눠 관리하기 위한 기반입니다.
- 설정 화면에서 시스템 프롬프트, 다운로드 동작, P-GPT 자격 증명을 다룰 수 있게 확장했습니다. 사용자가 설정 파일을 직접 열어 고치는 일을 줄이려는 방향입니다.
- 설치와 런타임 준비 과정을 보강했습니다. 로컬 설치 후 웹 UI를 바로 켰을 때 빠진 준비 작업 때문에 실패하는 일을 줄이기 위한 작업입니다.
- 프로젝트 파일 목록과 라벨을 정리했습니다. 너무 넓은 범위의 파일이 노출되거나 이름이 헷갈리면 선택 실수가 생기기 쉬워서, 더 안전하고 알아보기 쉬운 표시로 바꿨습니다.
- 접근이 막히는 웹사이트를 조사할 때 일반 검색이나 단순 가져오기로 끝나지 않도록 `insane-search` 스킬을 추가했습니다. 403/차단 페이지, X/Twitter, Reddit, YouTube, Naver, GitHub, Medium 같은 사이트에서 여러 우회 경로와 공개 API, 실제 브라우저 렌더링까지 시도할 수 있는 조사용 능력을 넣은 것입니다.
- 디자인 결과물을 더 잘 만들기 위해 `huashu-design` 스킬 묶음을 도입하고 영어 설명도 정리했습니다. 웹 UI가 단순 채팅 화면을 넘어, 프로토타입, 시각 자료, 슬라이드형 결과물까지 만들 수 있는 방향으로 확장될 수 있게 하려는 시도였습니다.
