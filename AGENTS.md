# MyHarness 작업 지침

- 사용자에게는 존댓말을 사용하세요.
- MyHarness 프리뷰용 단일 HTML 산출물을 만들 때는, 설치나 빌드 없이 바로 확인하는 목적이면 CDN 기반 라이브러리 사용도 선택지로 고려하세요. React, ReactDOM, Chart.js, ECharts, Three.js, Lucide, Tailwind CDN 등이 필요할 수 있습니다.
- CDN은 강제하지 마세요. 순수 HTML/CSS/JS로 충분하면 의존성을 늘리지 말고, 장기 유지보수용 앱이나 복잡한 프로젝트 구조가 필요하면 Vite/Next 같은 번들러 기반 구성을 고려하세요.
- 단일 HTML 프리뷰에서는 가능한 한 CSS와 앱 코드를 한 파일 안에 모아 file card에서 바로 열리게 하세요. 외부 파일을 상대 경로로 나누면 `iframe srcdoc` 프리뷰에서 경로 문제가 생길 수 있습니다.
- 단일 HTML 산출물을 새로 만들 때는 `index.html`처럼 성의 없는 기본 이름을 쓰지 말고, 요청 내용을 반영한 짧은 kebab-case 파일명을 우선 사용하세요. 예: `tetris-game.html`, `sales-dashboard.html`, `github-trends-report.html`.
- 단, 사용자가 `index.html`을 명시했거나 기존 앱의 진입점을 유지해야 하는 경우에는 `index.html`을 사용할 수 있습니다.
- 사람이 읽는 문서 파일을 새로 만들 때는 가능한 한 한국어를 사용해 보고서명/문서명을 정하는 것을 권장하세요.
- UI를 만들거나 수정할 때는 가능한 한 슬림하고 깔끔한 밀도를 선호하세요. 버튼, 카드, 모달, 선택지의 모서리 round는 과하지 않게 절제하고, 불필요하게 두툼한 패딩이나 큰 행 높이는 피하세요.
- UI 툴팁은 브라우저 기본 `title` 툴팁을 쓰지 말고, 앱의 기존 커스텀 툴팁 스타일과 동작을 재사용하세요. 기본 툴팁은 표시 지연과 디자인 불일치가 있어 MyHarness 화면에서는 피하세요.
- 스킬 설명 번역은 원본 `SKILL.md`가 아니라 UI i18n/매핑에서 처리하세요.
- UI 화면 점검용 stdout/stderr 로그, 스크린샷, 임시 브라우저 프로필은 저장이 필요할 때 프로젝트 루트에 흩뿌리지 말고 `.myharness/ui-checks/` 같은 ignore된 로컬 작업 폴더에 모으세요.
- `PATCH_NOTES.md`는 하루에 한 번 수준의 변경 기록 정리용으로 보고, 사용자가 요청했거나 릴리즈/일일 정리처럼 명확히 필요한 경우에만 수정하세요.

## Karpathy Guidelines

- 코딩, 리뷰, 리팩터링 작업에서는 Andrej Karpathy식 간결한 엔지니어링 지침을 따르세요. 이 지침은 속도보다 신중함과 정확성을 우선합니다.
- 사용자에게 진행 상황을 설명할 때 특수한 스킬을 사용하는 경우에는 알려도 좋지만, 대부분의 코딩 작업에 기본 적용되는 `karpathy-guidelines`는 굳이 언급하지 말고 실제로 할 작업만 자연스럽게 말하세요.
- 구현 전에 가정을 명시하세요. 해석이 여러 가지이거나 요구가 불명확하면 조용히 추측하지 말고 혼란스러운 지점을 드러내세요.
- 요청받은 범위 안에서 문제를 해결하는 최소한의 코드를 작성하세요. 단일 사용처를 위한 추상화, 요청되지 않은 확장성, 불필요한 설정 옵션을 추가하지 마세요.
- 기존 코드를 수정할 때는 필요한 부분만 외과적으로 변경하세요. 주변 코드 정리, 무관한 리팩터링, 기존 스타일 변경은 사용자가 요청하지 않는 한 하지 마세요.
- 변경한 코드 때문에 새로 생긴 미사용 import, 변수, 함수는 정리하되, 기존에 있던 무관한 죽은 코드는 삭제하지 말고 필요하면 언급만 하세요.
- 작업 목표를 검증 가능한 기준으로 바꾸세요. 버그 수정은 재현 테스트 후 통과시키고, 검증 가능한 명령이나 확인 절차로 완료 여부를 확인하세요.
